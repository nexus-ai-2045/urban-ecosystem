# Cross-world Operator Mode Roadmap

status: draft
owner: manager
updated: 2026-06-08

## Purpose

Cross-world Operator Mode is a roadmap for letting a viewer or operator
temporarily inhabit, inspect, and guide agents across multiple simulation
layers without changing the existing Urban Ecosystem data contract in the
first roadmap PR.

This document is public-facing. External fictional works and character names
used during ideation are intentionally not used as implementation identifiers.
Public names below are abstract archetypes and milestones only.

## Roadmap

1. **Sentinel MVP**: Define the smallest operator mode for selecting an agent,
   entering an inspection/control viewpoint, and returning to normal replay.
   The wake phrase remains an internal-only placeholder until human review
   approves public wording.
2. **World Bridge Actor**: Add a conceptual state model for moving between
   `physical`, `simulated`, and `liminal` layers.
3. **Guide / Partner**: Define assistant-like guide and companion archetypes
   for explaining what the operator is seeing and helping with safe actions.
4. **Agent Roster**: Expand monitoring, pursuit, and intervention archetypes
   without encoding copyrighted names or real-person identities.
5. **Cyber-ops Milestones**: Sketch cross-world operations, field support, and
   supervisor roles as future work orders.
6. **Equivalent Exchange Pair**: Explore cost, restoration, pair dependency,
   and body/soul separation as abstract social-simulation mechanics.
7. **Pillar Council Arc**: Model guardian, council, patron-leader, and
   corruption-prime structures for social hierarchy and conflict.
8. **Unstable Power Arc**: Explore city-scale instability, uncontrolled power,
   awakening events, and containment dynamics.
9. **Boundary War Arc**: Explore wall/boundary logic, external-world pressure,
   protector behavior, strategist behavior, and elite intervention.
10. **Fighter Archetype Set**: Add discipline, rivalry, precision, training,
    and event-duel concepts as optional scenario patterns.
11. **Social-Tech Mirror Lab**: Use short scenarios to examine how technology
    changes human behavior, incentives, identity, and trust.
12. **Judgment Game Arc**: Explore judgment, surveillance, inference,
    counter-inference, and adversarial reasoning.
13. **Ecological Mediation Arc**: Explore environmental negotiation,
    collective organisms, swarm intelligence, and non-human agency.
14. **Pilot Sync Arc**: Explore a reluctant pilot, quiet mirror, rival pilot,
    synchronization threshold, and bio-machine world pressure as abstract
    operator-entry mechanics.
15. **Human/AI Evaluation Lab**: Design a non-deceptive evaluation layer for
    human-likeness, intention inference, boundary recognition, and evaluator
    calibration.
16. **Three-world Layer**: Treat `physical`, `simulated`, and `liminal` as the
    initial three-layer world model.
17. **Event + Music Layer**: Capture vehicle-event and virtual-performer
    participation patterns, then prototype cheap 8-bit WebAudio cues in a
    later implementation work order.
18. **Next Motif Expansion Slot**: Keep a standing extension point for the
    next approved motif, so new archetypes can be adopted without rewriting
    the roadmap structure.
19. **Post-Singularity Scenario Boundary**: Treat post-singularity simulation
    as bounded scenario families, not prediction. The roadmap can explore
    worlds where capability growth, identity, governance, embodiment, and
    human/AI boundaries change, but it must not claim to forecast real
    post-singularity outcomes.
20. **Chaotic Three-Body World Benchmark**: Use a public-safe three-body
    dynamics benchmark for scenarios where agents must adapt to unstable eras,
    short-lived stability windows, and sensitive dependence on initial
    conditions. This benchmark tests whether the roadmap can separate
    predictable local policy from unpredictable long-run world state.
21. **Frontier AI Capability Layer Benchmark**: Benchmark the top capability
    layer as a scenario family, not a leaderboard. The benchmark should use
    public frontier-safety concepts such as long-horizon autonomy, tool use,
    cybersecurity uplift, CBRN-adjacent misuse risk, manipulation, loss of
    control, self-improvement, and governance thresholds.
22. **Future-Known Implementation Frame**: Compare older future-world images,
    current local implementation capacity, and the concrete gap this repo can
    close. This is not prophecy; it is a review method for asking whether a
    motif is now implementable with ordinary local tools.
23. **Human Time-Sense Boundary Benchmark**: Test whether agent reports expose
    wait value, progress, stuck state, next return point, and embodied urgency
    in a way that humans can understand. Raw intelligence is not treated as
    the only human/AI boundary.
24. **Numeric Operating Protocol**: Keep a speculative protocol slot for
    cycle, compression, opening, and split-count rules. Numeric meaning is a
    creative operating hypothesis only, not mathematical or religious truth.
25. **FDE Packet Router Benchmark**: Use an `entry -> packet -> evidence ->
    decision -> closure` loop as a benchmark for whether roadmap ideas can be
    serialized, reviewed, parked, routed, or rejected without becoming loose
    chat residue.
26. **Agent Harness Layer Benchmark**: Separate model capability from product
    harness quality: goals, permissions, approval UX, tool routing, recovery,
    evals, CLI/IDE coordination, and workspace memory.
27. **User Oversight Fourth-Power Layer**: Define the user as the external
    observer and fourth-power media layer: the role that can inspect reports,
    question framing, demand evidence, publish or withhold publication, and
    counterbalance internal agent consensus.
28. **Deliberative Separation-of-Powers Layer**: Split roadmap decisions into
    proposing, reviewing, and execution branches, with final user oversight.
    Major changes should pass through a council-style decision record before
    implementation.
    The default shape is three AI branches plus one meta-user oversight layer.
29. **Repository-as-Skill Mesh**: Treat the repo as a collection of callable
    skills and recursive references. A future implementation may expose skill
    entrypoints, skill-call records, dependency links, and self-reference
    checks, while keeping code, docs, and governance layers distinct.
30. **P2P Distributed Operations Spike**: Explore whether peer-to-peer
    distribution belongs in this project at all. The first step is a design
    spike for identity, trust, sync, conflict resolution, offline behavior,
    moderation, and abuse resistance, not a network implementation.
31. **Cloud Capacity Envelope**: Document where cloud services could help with
    storage, simulation, evaluation, or batch processing, while keeping all
    cloud, secret, quota, billing, deployment, and external API actions behind
    explicit human gates.

## Expansion Guarantees

Every future motif adoption must include both of the following:

- **Archetype guarantee**: at least one actor or role archetype that can appear
  in agent selection, operator guidance, conflict, evaluation, or replay
  explanation.
- **World guarantee**: at least one minimal world rule, pressure, layer,
  institution, ecology, machine system, or boundary condition. Character-only
  additions are not enough for this roadmap.

Operator-entry phrases are allowed as internal design inputs, but public code
must not hard-code copyrighted catchphrases, character names, or quoted lines.
If phrase-triggered entry is implemented later, the public implementation should
refer only to abstract trigger classes such as `resistance_phrase` or
`identity_assertion`, with any sensitive wording held outside public code and
approved by human review.

## Minimum World Richness Guarantee

Every approved motif or benchmark must include enough world context to make the
scenario feel lived-in rather than character-only. The minimum world packet is:

- **Place and environment**: where the scenario happens, what geography,
  climate, built environment, ecology, or spatial constraint matters.
- **Rules of possibility**: what physics, technology, magic-like system, AI
  capability boundary, or simulation rule changes what agents can do.
- **Social fabric**: who belongs to the world, what cultures, institutions,
  councils, factions, households, or communities shape behavior.
- **Resources and power**: what is scarce, who controls access, and what
  economic, political, or infrastructural pressure follows from that.
- **History and memory**: what past event, myth, failure, archive, trauma, or
  inherited premise still constrains the present.
- **Daily life signal**: one ordinary object, ritual, job, route, meal, media
  habit, interface, or local custom that makes the world inspectable at street
  level.
- **Change pressure**: what unstable force, crisis, frontier capability,
  ecological pressure, or social contradiction is making the world move.

If an addition cannot provide all seven fields yet, it can enter the roadmap
only as a `parking-lot` or `watch` item. It is not ready for implementation
until the missing fields are either filled or explicitly waived by human
review.

## Add Request Intake

Roadmap additions can be proposed from GitHub or Discord when a maintainer or
participant says an equivalent of "add this" for a motif, archetype, or world
idea. GitHub remains the public source of truth; Discord is only an intake and
discussion surface while the public collaboration channel is frozen or lightly
used.

An add request is eligible for the GitHub roadmap only after review confirms:

- The request can be renamed into public-safe abstract archetypes.
- The request includes, or can be paired with, a minimal world guarantee.
- The roadmap addition does not include secrets, internal URLs, private user
  data, copyrighted character names as implementation IDs, or quoted lines as
  public code triggers.
- The change lands through a GitHub docs update, GitHub issue, or PR reviewed
  under the TYPE1 public PR gate.

The minimum GitHub roadmap record for an approved add request is:

- public roadmap name
- source surface: `github` or `discord`
- archetype guarantee
- world guarantee
- minimum world packet status
- next work-order or issue draft
- human reviewer or review note

Do not auto-write from Discord chat into GitHub. If automation is added later,
it must create a private or local draft first and wait for human review before
creating any public GitHub artifact.

## Non-goals And Lifecycle Rules

This roadmap deliberately does not guarantee that every proposed motif becomes
an implementation task. It also does not guarantee zero orphan ideas. An
orphan idea means an approved roadmap entry that has no next issue, work order,
owner, reviewer, or explicit parking-lot note.

The acceptable orphan threshold is:

- `0` orphan ideas for Sentinel MVP and any active implementation milestone.
- Up to `2` orphan ideas, or up to `10%` of approved roadmap entries,
  whichever is lower, for exploratory future milestones.
- Any orphan idea above that threshold must be moved to a parking lot,
  paired with a world guarantee, or converted into a GitHub issue / work-order
  draft after human review.

Stale status should be self-reported first. Each roadmap item can carry a
future status note such as `fresh`, `watch`, `stale-self-reported`, or
`parked`. A stale item is not a failure; it is a signal that the motif,
world rule, owner, or next action needs review before implementation.

Heartbeat checks may be used later to ask for stale self-reporting, but they
must stay read-only or draft-only by default:

- They may remind maintainers to review old roadmap items.
- They may prepare local/private stale reports.
- They may not auto-close, auto-delete, auto-publish, or auto-create public
  GitHub / Discord artifacts.
- They may not mark an item ready for implementation without human review.

The roadmap also does not attempt to reproduce fictional characters, quote
protected lines, predict real people, or turn inspiration notes into public
implementation IDs.

Post-singularity scenarios are allowed only as speculative world-rule tests.
They must state their assumptions, uncertainty, failure modes, and what would
make the scenario invalid. They should compare multiple possible worlds rather
than presenting one future as inevitable.

The chaotic three-body benchmark is allowed as a model case because it has an
external scientific basis and a clear simulation boundary. It should benchmark:

- initial-condition sensitivity
- stable-window vs chaotic-era classification
- agent adaptation under unreliable long-run forecasts
- whether social policies can be evaluated locally even when global prediction
  fails

It must not cite or reproduce fictional story details. The public benchmark is
about the mathematical three-body problem and chaos, not a protected narrative
setting.

The frontier AI capability benchmark is allowed only as a reviewable benchmark
family. It must not rank current labs or models, claim private capability
knowledge, run live dangerous-capability tests, or publish exploit-like
instructions. The benchmark should instead ask whether a scenario has enough
structure to compare:

- long-horizon autonomous task completion
- tool-use and computer-use escalation
- cyber-defense vs cyber-offense boundary pressure
- biological, chemical, nuclear, or radiological misuse-adjacent constraints
- persuasion, manipulation, and belief-change pressure
- autonomous replication, resource acquisition, or safeguard-undermining risk
- self-improvement or AI R&D acceleration pressure
- human review, incident response, and deployment-governance gates

Any future implementation must use toy tasks, red-team-safe abstractions, or
public evaluation summaries rather than operational misuse procedures.

Local dialogue and Obsidian-derived candidates are allowed as internal source
inputs, but they must be converted into public-safe roadmap language before
publication. Public docs must not expose private note paths, private chat
phrases, voice-chat transcripts, or unreviewed quoted source text. These
candidates should normally be labeled `project-hypothesis` or
`creative-hypothesis` until a reviewer ties them to repo policy or external
practice.

The fourth-power layer means the user is the external monitor, not another
agent. Agents may propose and deliberate, but the user can interrupt, question
evidence, change priority, require public-safety review, or refuse publication.
This keeps media-like oversight outside the internal agent system.

The deliberative separation-of-powers layer should avoid single-agent
unilateral decisions for major roadmap changes:

- **Proposal branch**: drafts a candidate, source category, and world packet.
- **Review branch**: checks public safety, provenance, tests, and missing
  assumptions.
- **Execution branch**: edits docs, opens approved work orders, or implements
  only after the required gates.
- **Fourth-power oversight**: the user can challenge any branch and is the
  final public-release authority.

This creates the default fractal decision shape for the roadmap: three AI
branches plus one meta-user oversight layer. The same shape can be applied at
multiple scales: roadmap item, work order, implementation PR, release review,
or public collaboration decision. At each scale, the branches can recurse into
smaller proposal/review/execution packets, but the meta-user layer remains
outside the agent system and can stop or redirect the loop.

FDE is the adopted decision protocol for this fractal structure. Each recursive
decision should be packetized before it grows:

- **Entry**: what triggered the decision.
- **Packet**: the smallest reviewable unit.
- **Evidence**: local files, tests, sources, or explicit assumptions.
- **Decision**: adopt, revise, park, reroute, reject, or escalate.
- **Closure**: what changed, what remains, and who can reopen it.

FDE recursion must include loop guards. A branch may not keep expanding
indefinitely; it must either close, park, escalate to the meta-user, or split
into a new explicit work order.

The repository-as-skill mesh is a future architecture hypothesis, not part of
the docs-only PR. It should be explored through small, reviewable records:
skill entrypoint, allowed inputs, allowed outputs, dependencies, caller,
evidence, and closure status. Recursive skill calls must include loop guards
and a maximum traversal depth before any implementation.

P2P distribution is intentionally a spike because the project does not yet
define identity, trust, transport, data ownership, sync semantics, moderation,
or abuse handling. A future P2P design should answer at least:

- What is shared between peers, and what must stay local?
- How are peers identified, authenticated, and revoked?
- How are conflicts resolved when two peers update the same world state?
- What is the fallback when a peer disappears or sends stale data?
- How are spam, unsafe payloads, and malicious peers contained?
- Which parts are better served by a central service until the trust model is
  mature?

Cloud services can be used in future work only as a capacity envelope. The
current roadmap PR does not run cloud commands or create cloud resources. Any
future cloud use must name the target service, account/project, data that would
leave the local machine, cost/quota risk, credentials involved, and rollback
plan before execution.

## Reviewability And Provenance

Each roadmap item should be reviewable before it becomes implementation work.
The source category must be explicit:

| Source category | Meaning | Review handling |
| --- | --- | --- |
| `public-policy` | Existing project policy, gate, or public collaboration rule. | Verify against repo docs before implementation. |
| `external-practice` | Grounded in external product or community-management practice. | Keep a short source link or review note. |
| `project-hypothesis` | A local operating rule chosen for this project. | Treat as adjustable until tested in a real review cycle. |
| `creative-hypothesis` | Original motif synthesis or speculative design. | Do not present as industry practice; require human review before implementation. |

Best-practice grounding for this roadmap:

- GitHub labels and milestones are the preferred public categorization and
  tracking surface for roadmap work.
- Linear stays useful for internal planning, especially when high-level
  timeline/project views are separated from granular issue implementation.
- Stale automation is allowed only as a future draft/reminder mechanism. Public
  write behavior must remain disabled or human-reviewed; auto-close should stay
  off for motif-roadmap work.
- Discord intake must follow Discord platform rules and should not bypass the
  GitHub review path.

Potentially original or speculative parts of this roadmap include the archetype
names, world guarantees, orphan threshold, stale self-reporting vocabulary,
three-world model, event/music layer, post-singularity scenario boundary, and
the way this roadmap combines frontier AI safety concepts into a simulation
benchmark family. These are intentionally reviewable hypotheses, not claims of
established practice.

Before a future PR implements any item from this roadmap, reviewers should be
able to answer:

- What is the source category?
- What is the public-safe name?
- What is the archetype guarantee?
- What is the world guarantee?
- Does the minimum world packet cover place, rules, society, resources,
  history, daily life, and change pressure?
- Does the item include the user oversight role when public or external action
  is possible?
- Does the decision need proposal, review, and execution separation?
- Does the decision follow the three-AI plus meta-user fractal shape, or does
  it need a documented exception?
- Was the decision packetized through FDE before recursive expansion?
- If it is a recursive skill or P2P/cloud candidate, what loop, trust, cost, or
  external-action boundary applies?
- What is the next issue, work order, owner, reviewer, or parking-lot note?
- Is this still fresh, self-reported stale, or parked?
